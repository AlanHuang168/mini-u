# 预备知识：三个词，先听懂

| 词 | 人话解释 | 生活类比 |
|----|---------|---------|
| **大模型 / LLM** | 会聊天、会写文章的 AI（如 ChatGPT、DeepSeek） | 一个博学的「实习生」 |
| **API** | 程序之间打电话的「电话号码」 | 点外卖的下单接口 |
| **API Key** | 打这个电话要用的「密码」 | 你的外卖账号密码 |

<div class="warn">

</div>

<!--
口播：开始前先扫清三个术语。大模型就是像 ChatGPT 那样会写字的 AI，你可以把它想象成一个很博学但需要你给指令的实习生。API 是程序之间互相调用的接口，就像你用手机 App 下单外卖。API Key 是调用时的密码，而且是要花钱的——这点后面会专门讲一个真实的安全事故。

# 核心思想：把大事拆成小步

想象你要做一道菜 🍳，你不会「一步做完」，而是：

```
买菜  →  切菜  →  下锅炒  →  装盘
```

我们的 AI 应用也一样，叫做「**流水线 / Pipeline**」：

```
采集数据  →  AI 分析  →  生成报告
 (买菜)       (炒菜)       (装盘)
```

<div class="tip">

**每一步只干一件事** —— 这样出了问题，一眼就知道是哪一步坏了。

</div>

<!--
口播：这是整节课最重要的思想。做复杂的事情，不要想着一步登天，而要拆成一个个简单的小步骤，像流水线一样串起来。做菜要「买菜、切菜、炒菜、装盘」，我们的程序就是「采集数据、AI分析、生成报告」。每一步单独简单，串起来就完成了复杂任务。
-->

---

# 系统全貌：像一栋楼分层

```
┌─────────────────────────────┐
│  界面层：用户点点点的网页     │  ← 你看到的
├─────────────────────────────┤
│  功能层：单股分析/多股对比…   │  ← 不同的功能
├─────────────────────────────┤
│  流水线：采集 → 分析 → 报告   │  ← 干活的核心
├─────────────────────────────┤
│  底层：数据源 / AI / 缓存     │  ← 基础设施
└─────────────────────────────┘
```

<div class="tip">

**分层的好处**：换了顶楼的装修，不影响地基。改界面不用动核心逻辑。

</div>

<!--
口播：整个系统像一栋楼，分成几层。最上面是用户看到的网页界面，中间是各种功能，再下面是干活的流水线，最底下是数据源和AI这些基础设施。分层的最大好处是「互不干扰」——我想换个好看的界面，完全不用碰下面算分析的代码。
-->

# 第 1 步：采集数据 + 打分

**做两件事**：

1. 去网上抓这只股票的**行情**（股价、涨跌）
2. 用「**规则**」给它**打个分**（满分 100）

打分规则长这样（人人都看得懂）：

```yaml
股价 > 1000 元   →  加 15 分
今天涨了         →  加 10 分
ROE > 20%       →  加 20 分
```

<div class="tip">

规则写在**配置文件**里，改分数不用改程序 —— 运营同学也能维护。

</div>

<!--
口播：流水线第一步，采集数据。程序自动去网上抓股价、涨跌这些信息，然后用一套「规则」打分。规则很简单：股价高就加分、上涨就加分。关键是——这些规则不是写死在代码里，而是放在一个人人能读懂的配置文件里，改分数不用懂编程。
-->

---

# 为什么要「规则」而不全交给 AI？

<div class="warn">

大模型（AI）很聪明，但它**不擅长精确算数**，还可能「一本正经地胡说」（叫**幻觉**）。

</div>

<div class="tip">

规则引擎**算得准、可复现、能解释**「为什么是 65 分」。

</div>

**所以我们让它们分工**：

| 谁 | 负责什么 | 类比 |
|----|---------|------|
| **规则** | 算分数（精确的活） | 计算器 |
| **AI** | 写分析（表达的活） | 作家 |

<!--
口播：有同学会问，都有AI了，为什么还要写规则？因为AI有两个毛病：一是不擅长精确算数，二是会「幻觉」——一本正经地编造。而规则引擎算得准、能解释清楚为什么打这个分。所以我们让它们分工。
-->

---

# 🌟 金句：本课最重要的一句话

<br>

# 让规则引擎做**数学题**
# 让大模型做**作文题**

<br>

<div class="tip">

把「精确计算」交给代码，把「语言表达」交给 AI —— 这是做**可靠** AI 应用的核心原则。

</div>

<!--
口播：这句话请大家记下来，它是做AI应用的黄金法则：让规则引擎做数学题，让大模型做作文题。精确的、不能出错的部分交给代码；需要灵活表达、写得好看的部分交给AI。这样既发挥了AI的长处，又避开了它的短处。
-->

---

# 第 2 步：AI 分析

把第 1 步的**数据和分数**打包，交给 DeepSeek（一个国产大模型）：

```
你是一位资深股票分析师。
这是数据：{茅台，1190元，评分65…}
请写一份 250-400 字的专业分析，
最后给出评级：强烈推荐/推荐/观望/回避
```

AI 就会把冰冷的数字，翻译成**有逻辑、会说人话**的分析报告。

<div class="tip">

给 AI 的这段指令叫「**Prompt（提示词）**」—— 写好它是一门手艺。

</div>

<!--
口播：第二步，把数据和分数交给大模型DeepSeek。我们给它一段指令，告诉它「你是分析师，这是数据，请写一份分析」。这段指令有个专业名字叫Prompt，也就是提示词。同样的数据，Prompt写得好不好，AI输出的质量天差地别，所以「写提示词」现在是一门专门的手艺。
-->

---

# 提示词的小技巧

我们的提示词里明确要求：

- ✅ **结构**：基本面 → 风险 → 逻辑 → 建议
- ✅ **字数**：250-400 字（不能太长太短）
- ✅ **给选项**：评级只能从「强烈推荐/推荐/观望/回避」里选

<div class="tip">

**越具体，AI 越听话**。别只说「帮我分析」，要说清楚「结构、长度、格式」。

</div>

<!--
口播：写提示词有个诀窍：越具体越好。你不能只说「帮我分析一下」，那AI会天马行空。你要告诉它结构是什么、字数多少、评级只能从哪几个里选。把要求框得越清楚，AI的输出就越稳定可控。这个技巧大家平时用ChatGPT也能马上用上。
-->

---

# 第 3 步：多位大师来点评

同一只股票，让 AI **扮演不同的投资大师**来点评：

| 大师 | 风格 | 关注点 |
|------|------|--------|
| 巴菲特 | 价值投资 | 护城河、安全边际 |
| 赵老哥 | 游资短线 | 板块热度、情绪 |
| 木头姐 | 成长创新 | 颠覆性、增长 |
| 陈小群 | 宏观策略 | 政策、行业周期 |

<div class="tip">

每个大师就是一张「**人设卡**」（一个小配置文件）。加新大师 = 加一张卡，**不用改程序**。

</div>

<!--
口播：第三步很有意思。同一只股票，我们让AI分别扮演巴菲特、赵老哥这些风格迥异的投资大师来点评，得到多个不同视角的观点。每个大师就是一张「人设卡」，是个几行的小文件。你想加一个新大师，只要复制一张卡改改内容，完全不用碰程序代码。这就是「数据驱动」的威力。
-->

# 省钱小能手：缓存

<div class="warn">

**每次调用 AI 都要花钱、还要等几秒。** 同一只股票反复分析，太浪费！

</div>

解决办法：**缓存（Cache）**

```
有人问茅台  →  先看「小本本」上有没有记过
  记过且没过期  →  直接念出来（不花钱！）
  没记过        →  真的问 AI，然后记到小本本
```

<div class="tip">

缓存有「**保质期**」（本项目 6 小时）—— 因为股价会变，太旧的不能用。

</div>

<!--
口播：调用AI是要花钱的，而且要等待。如果同一只股票几分钟内被问了十次，每次都真去问AI，就是白白烧钱。所以我们加了「缓存」——像个小本本，问过的答案先记下来，下次直接念，不用再花钱问。但缓存有保质期，因为股价会变，太旧的答案就不能用了，我们设的是6小时。
-->

# 🛡️ 真实事故 1：密码泄露

**背景**：这个项目上线后，发现一个安全问题——

<div class="warn">

用户 A 输入了自己的 API Key（花钱的密码），
结果**用户 B 打开网页，也能用 A 的密码**！= 花 A 的钱 😱

</div>

**原因**：程序错误地把密码存在了「**所有人共享**」的地方。

**修复**：改成「**每个人一个独立小房间**」，互相看不见。

<!--
口播：现在讲最宝贵的部分——真实踩过的坑。这个项目上线后发现：用户A输入了自己的付费密码，结果用户B打开网页居然也能用A的密码，等于花A的钱。这是很严重的安全事故。原因是程序把密码存错了地方，存到了所有人共享的区域。修复方法是给每个用户一个独立的「小房间」，互相隔离。
-->

---

# 事故 1 的教训

<div class="warn">

**敏感信息（密码、密钥）绝不能存在「大家共享」的地方。**

</div>

**类比**：
- ❌ 把银行卡密码写在**公司公告栏** → 人人可见
- ✅ 把密码放进**自己的保险箱** → 只有你能开

<div class="tip">

做多用户的网站，一定要问自己：**「这个数据，别的用户会不会看到？」**

</div>

<!--
口播：这个事故的教训是：密码这类敏感信息，绝不能存在大家共享的地方。就像你不会把银行卡密码贴在公司公告栏上。做任何多用户的网站，都要养成习惯问自己：我存的这个东西，别的用户会不会看到？
-->

# 🛡️ 真实事故 2：神秘崩溃

**现象**：同一份代码——

- 在服务器上（Linux）✅ 正常
- 在电脑命令行 ✅ 正常
- 但用网页跑 ❌ **直接崩溃**

**排查思路**（很重要的方法论）：

<div class="tip">

对比找不同：**操作系统？运行方式？** 把可疑的部分**单独拎出来测**，缩小范围。

</div>

最后定位：某个底层组件在特定环境下**不兼容** → 加了个「演示模式」绕过它。

<!--
口播：第二个坑更玄乎。同样的代码，服务器上正常，命令行正常，就是网页一跑就崩溃。怎么排查？核心方法是「对比找不同、单独拎出来测」。我们发现区别在于运行方式，最后定位到一个底层组件在特定环境下不兼容。解决办法是加了个「演示模式」开关绕过它。这个「对比+隔离」的排查思路，你以后遇到任何bug都用得上。
-->

---

# 事故 2 的教训

<div class="tip">

**「在我电脑上好好的呀」** —— 是程序员最常说、也最坑的一句话。

</div>

同样的代码，换个环境就可能出问题，因为环境有差异：

- 操作系统不同（Windows / Mac / Linux）
- 软件版本不同
- 运行方式不同

**方法论**：出 bug 别慌，**对比环境差异 + 逐步缩小范围**。

<!--
口播：这个坑的教训，也是程序员那句经典甩锅——「在我电脑上好好的呀」。同样代码换个环境就出问题，太常见了，因为操作系统、软件版本、运行方式都可能不一样。所以遇到bug不要慌，冷静地对比环境差异，一步步缩小范围，总能找到根源。
-->

# 🛡️ 真实事故 3 & 4：小疏忽

**事故 3：忘了声明依赖**
程序用到一个工具包，却忘了写进「购物清单」（requirements.txt）。
👉 **教训**：程序需要什么，必须**明明白白列出来**，不能靠「碰巧装了」。

**事故 4：把巨大的临时文件传上去了**
一不小心把「虚拟环境」（几百 MB 临时文件）提交到了代码仓库。
👉 **教训**：临时文件、密钥文件**不该进仓库**，要写进「黑名单」（.gitignore）。

<!--
口播：还有两个小疏忽。一个是忘了声明依赖——程序要用某个工具包，却没写进清单，换台电脑就找不到。教训是：程序需要什么，必须白纸黑字列清楚。另一个是不小心把几百MB的临时文件传到了代码仓库。教训是：临时文件和密钥文件不该进仓库，要提前写进「黑名单」。这些都是新手最常犯的小错误。
-->

# 复盘：一次完整的分析之旅

```
输入「600519」
   │
   ▼ ① 采集：抓行情 + 规则打分 → 65分
   ▼ ② AI分析：把数据交给DeepSeek → 写出报告
   ▼ ③ 多角色：4位大师分别点评
   ▼ ④ 生成漂亮的网页报告 + 保存
   │
   ▼ 你看到完整报告 ✅
```

<div class="tip">

数据的旅程：**数字 → 文字 → 网页**。几乎所有 AI 应用都是这个套路。

</div>

<!--
口播：我们把整个流程串一遍。你输入600519，第一步采集数据打分得65分，第二步交给AI写分析，第三步请4位大师点评，第四步拼成网页报告并保存。注意数据的形态变化：从数字，变成文字，最后变成网页。你以后看到的绝大多数AI应用，都是这个套路。
-->

# 总结：Mini-U 教会我们 5 件事

1. 🧩 **拆解** —— 把复杂任务拆成流水线的小步骤
2. 🤝 **分工** —— 规则做数学题，AI 做作文题
3. ⚙️ **配置驱动** —— 规则/角色/提示词都外置，改配置不改代码
4. 💰 **工程优化** —— 缓存省钱、出错兜底
5. 🛡️ **安全意识** —— 密钥隔离、环境差异、别踩坑

<div class="tip">

**AI 应用 ≠ 把问题丢给 AI**，而是用工程手段，把 AI 编排进一条可控的流水线。

</div>

<!--
口播：最后总结，这个小项目教会我们5件事：学会拆解、懂得分工、配置驱动、工程优化、安全意识。请记住最后这句话：做AI应用，不是简单地把问题丢给AI就完事，而是用工程的手段，把AI这个聪明但不完全可靠的伙伴，编排进一条你能掌控的流水线里。
-->
